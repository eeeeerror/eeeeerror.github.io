# 一些陌生的项目

# 1."UsedBytesInTheLastPage"

是 Windows 操作系统中的一个系统信息项，表示最后一页（页大小通常为 4KB）中实际使用的字节数。

这个信息可以通过系统调用 API 来获取，例如 GetProcessMemoryInfo 函数可以获取当前进程的内存使用情况，其中就包括了 UsedBytesInTheLastPage。在任务管理器中也可以查看到每个进程的 “最后一页的使用” 信息。

# 2."InitialSP"

通常指的是程序在执行之前栈顶指针的初始值，也称为初始堆栈指针。在一个程序的运行过程中，栈是用来存放局部变量、函数调用、函数返回值等信息的数据结构，而初始堆栈指针则是指向栈顶位置的指针。

在 x86 架构的 CPU 中，栈是由 CPU 硬件直接支持的，通常在程序加载时，操作系统会为该程序分配一段连续的内存空间作为栈空间。通过设置初始堆栈指针的值，可以指定程序的栈空间起始地址及其大小，在程序运行时，将栈顶指针不断移动，即可实现栈的正常使用。

在汇编语言中，可以通过设置 SP 寄存器（Stack Pointer，栈指针）的值来控制初始堆栈指针的值。

# 3."AddressOfNewExeHeader"

是 Windows PE 文件格式中 Optional Header 的一个字段，它存储了在内存中加载 PE 文件时，PE 头的位置信息。该字段指定了文件在被映射到内存后所使用的 PE 头的虚拟地址（VA），同时也用于确定代码执行的入口点。

在程序启动时，Windows 操作系统会将 PE 文件映射到进程的虚拟地址空间中，并创建一个进程环境块（PEB）以跟踪该进程的状态。在映射 PE 文件时，Windows 会从 PE 文件中读取 Optional Header 中的 AddressOfNewExeHeader 字段，把其值加上基址作为 PE 头的虚拟地址（VA），然后将该 VA 映射到进程的虚拟地址空间中。

在执行 PE 文件时，Windows 会根据 Optional Header 中的 ImageBase 字段加载并执行代码，该字段指定了 PE 文件映射到进程的虚拟地址空间中的基地址。执行过程中，Windows 通过 AddressOfEntryPoint 字段定位程序入口点，开始执行程序逻辑。

对于未映射到内存的 PE 文件，AddressOfNewExeHeader 没有意义。

如果修改了 PE 文件的 Optional Header 中的 AddressOfNewExeHeader 字段值，那么在文件被映射到进程的虚拟地址空间时，PE 头的实际地址就会被修改。这将导致 Windows 操作系统无法正确加载并执行程序，因为操作系统会试图在 PE 头的错误位置读取和解析 PE 头数据，从而导致程序崩溃或出现其他未知的行为。

此外，在 PE 文件映射到内存时，如果 PE 头的实际地址与加载器（如 rundll32.exe）所期望的地址不一致，会触发 Windows 中的 DEP 保护机制（Data Execution Prevention），导致程序无法正常执行。

因此，修改 PE 文件的 AddressOfNewExeHeader 字段值是非常危险的，并且很容易破坏程序的完整性和可靠性。只有在特定情况下、并且确切知道修改的含义和影响的情况下才应该进行此类操作。

# 3.5**“This program cannot be run in DOS”**

是一个常见的错误提示，通常出现在尝试在 MS-DOS 或早期版本的 Windows 中运行 16 位或 32 位的 Win32 应用程序（例如 Windows 应用程序或驱动程序）时。

其原因是这些应用程序依赖于 Windows 操作系统中的功能，而 MS-DOS 和早期版本的 Windows （如 Windows 3.x）并不支持这些功能。因此，在这些操作系统中尝试运行这些应用程序时，都会收到类似 “该程序无法在 MS-DOS 模式下运行” 的错误提示。

另一方面，Windows 操作系统中的 PE 文件格式也使用 “MZSignature” 标识和 DOS 头部，以支持早期的 MS-DOS 可执行文件格式，并向后兼容。当在 MS-DOS 或早期版本的 Windows 中尝试运行 PE 格式的可执行文件时，Windows 会显示 “该程序无法在 MS-DOS 模式下运行” 的错误提示，提示用户该程序需要运行在 Windows 操作系统中才能正常工作

# 4."AddressOfRelocationTable"

是 Windows PE 文件格式中可选头部（Optional Header）的一个字段，它保存了标记所有需要重定位的代码和数据段的修正项数组相对于映像基址（Image Base）的偏移量。Windows 在将 PE 文件加载到内存时，会根据这个字段指定的地址将修正项数组加载到内存中进行动态地址重定位。

PE 文件格式通常将代码、数据和重定位信息分开存储，其中重定位信息用来告诉操作系统在映射代码和数据段时，哪些地址需要进行修正。AddressOfRelocationTable 字段指定了修正项数组的起始地址，并通过 VirtualSize 字段指定了修正项数组的大小。

动态地址重定位是一种机制，在加载 PE 文件时动态修改代码和数据段的内存地址，确保它们在内存中都是有效的。这通常在 PE 文件被加载到一个不同的内存地址时需要执行，因为 PE 文件的绝对地址通常是固定的，不同的位置可能会导致代码和数据段地址失效。

如果 AddressOfRelocationTable 字段中的地址无效或修正项数组中的偏移量无法被应用到正确的地址上，则应用程序可能无法正确加载，导致运行错误或崩溃。

# 4.5"Image Base"

在 Windows PE 文件格式中，PE 头和可选头描述了 PE 文件的各种属性，其中可选头包括 Image Base 字段。Image Base 是一个指针，指向 PE 文件在内存中加载时所分配的基地址。当 PE 文件被映射到进程地址空间中时，使用该基地址来计算每个节在内存中所占用的地址范围。

PE 文件的 Image Base 值位于可选头的第 15 个字节处，对应于 IMAGE_OPTIONAL_HEADER 结构体中的 ImageBase 字段。在 32 位 PE 文件中，Image Base 通常为 0x00400000；在 64 位 PE 文件中，Image Base 通常为 0x0000000140000000。

需要注意的是，Image Base 值必须是一个页对齐的地址，因为 Windows 操作系统以页为单位进行内存映射，如果 Image Base 值不是页对齐的，则可能会导致性能损失或者其他问题。因此，在创建 PE 文件时，应该指定合适的 Image Base 值，并且确保它是一个页对齐的地址。

# 节表的一些奇奇怪怪的东西

# 1.rdata

是 PE 文件（Portable Executable File）中的一个节区（Section），用于存储只读数据，通常包括字符串、常量、程序中使用的各种资源等。这些数据在程序运行期间不能被修改，因此它们会被放在只读节区，以确保数据的安全性。

一般来说，.rdata 节区是由编译器自动生成的，开发者可以向其中添加常量和字符串等只读数据。在程序运行期间，这些数据可以通过指针进行访问，但是不能直接修改。如果程序需要修改这些数据，那么它们应该被放在其他节区中，例如 .data 节区或 .bss 节区。

需要注意的是，虽然 .rdata 节区中的数据是只读的，但是它们并不是加密的，因此这些数据仍然可以被恶意攻击者读取和利用。为了提高程序的安全性，在处理敏感数据时应该采取额外的安全措施，例如使用加密算法对数据进行加密，以防止数据泄露和篡改。

# 2.text

.text 是 PE 文件（Portable Executable File）中的一个节区（Section），用于存储程序的可执行代码。在程序运行期间，操作系统会把 .text 节区中的指令加载到内存中，并按照顺序逐条执行，从而实现程序的功能。

一般来说，.text 节区中的代码是由编译器生成的汇编代码，它们经过了链接器和优化器的处理，以提高程序的执行效率。对于不同的 CPU 架构和操作系统，.text 节区中的代码格式也不一样，例如 Windows 上的 x86 架构程序使用的是 PE 文件格式，而 Linux 上的 x86 架构程序使用的是 ELF 文件格式。

需要注意的是，虽然 .text 节区中的代码是只读的，但是它们并不是加密的，因此这些代码仍然可以被恶意攻击者读取和利用，例如反汇编、调试等操作。为了提高程序的安全性，在处理机密信息时应该采取更加安全的方式，例如使用加密算法对数据进行加密，或者在代码中添加防抄袭措施，以避免代码被盗用和篡改。

# 3.pdata

.pdata 是一种在 Windows 操作系统中用于存储异常处理程序信息的节 (section)。.pdata 节中包含了特定函数的代码范围和其异常处理器的指针，当发生异常时，程序可以通过.pdata 节中的信息来找到相应的异常处理器。该节通常出现在可执行文件 (.exe) 或动态链接库 (.dll) 中的代码段。

# 4.rsrc

.rsrc 是一种在 Windows 操作系统中用于存储异常处理程序信息的 (section)。.rsrc 节通常出现在可执行文件 (.exe) 或动态链接库 (.dll) 中，它包含了程序使用的各种资源，例如图标、对话框、菜单、位图、字符串等等。这些资源可以通过程序代码来访问，也可以通过应用程序编程接口 (API) 函数来访问。在 Windows 开发中，使用资源管理器 (Resource Editor) 工具可以方便地编辑.rsrc 节中的资源信息。

# 5.reloc

.reloc 是一种在 Windows 操作系统中用于存储地址重定位信息的节 (section)。在 Windows PE 文件中，代码和数据通常使用相对虚拟地址 (RVA，Relative Virtual Address) 来表示，但是当可执行文件或动态链接库被装载到不同的内存地址时，这些地址需要进行重新定位。.reloc 节中包含了程序需要重定位的各种地址信息，当程序加载到内存时，系统会根据.reloc 节中的信息重新计算地址，并对程序进行重定位，使得程序可以在新的内存地址上正确地运行。

# 6.idata

.idata 是一种在 Windows 操作系统中用于存储导入表 (import table) 信息的节 (section)。在可执行文件 (.exe) 或动态链接库 (.dll) 中，如果需要使用其他 DLL 库中的函数，就需要在程序中建立对这些 DLL 库的引用，并在运行时将它们加载到内存当中。.idata 节中包含了程序依赖的各个 DLL 库的名称、函数名称以及函数地址等信息。在程序运行时，系统会根据.idata 节中的信息，将所需的 DLL 库加载到内存中，并将相应的函数地址填充到程序代码中，使得程序可以正确地调用这些函数。

# 7.msvcjmc

msvcjmc 是一种 Microsoft Visual Studio 的组件，它提供了用于性能分析和调试的工具和库。该组件包含 Profile Guided Optimization (PGO) 和 CPU Sampling 功能。PGO 可以帮助开发人员优化应用程序的代码执行路径，从而提高程序的性能。CPU Sampling 功能则可以帮助开发人员识别并解决应用程序中的性能问题。msvcjmc 通常用于 Windows 开发环境中，可以与 Visual Studio 一起使用。

# 8.00cfg

.00cfg 是一种在 Windows 操作系统中用于存储控制流完整性 (CFI, Control Flow Integrity) 检测信息的节 (section)。CFI 是一种安全机制，可以防止攻击者利用控制流劫持漏洞 (control-flow hijacking vulnerability) 来改变程序的正常执行流程，从而实施恶意攻击。.00cfg 节中包含了程序在运行时对控制流的检测信息，包括函数调用、跳转目标、返回地址等等。当程序运行时，系统会根据.00cfg 节中的信息进行控制流的验证，确保程序的执行流程不受恶意攻击的影响。

# 重定位

1. 病毒的重定位技术

​ 病毒的重定位技术应该也算是最基本和最有用的技术之一了，其实重定位技术只要是对外壳开发比较熟悉的朋友都应该很熟悉了。重定位顾名思义重新定位地址。上篇文章中我们已经讲解到了由于我们被感染对象的不同，我们部分定义的变量被插入的地址也不同。所以这就需要我们重新定位，也就是今天的主题 —— 重定位技术。

​ 首先我们来看

假如我们定义一段变量

szText    db 'Virus Dels Demo', 0

​ 假设我们这个变量在我们病毒体的地址是 00403201h, 那么我们插入到被感染对象后这个变量的地址是 00408602h。那么我们如何能让我们的病毒体代码在运行时期来计算到被插入被感染对象后变量的地址呢？
​
​ 我们思索下是否能找到一些固定的地方。有时候换位思考是很不错的，我们病毒体既然被插入的地址是不固定的，那么病毒体的一些绝对偏移它相对病毒体的偏移肯定是固定的吧。
​
​ 假设一个病毒的起始位置是 00400021h, 那么它插入到被感染对象后的位置是 00500021h, 那么我们是不是只要知道一个变量相对于我们病毒起始位置的偏移就可以求得这个变量插入后的位置了？

​ 比如我们的 szText 变量相对于病毒起始位置的偏移是 9h, 那么只要通过求得病毒插入到被感染对象后的病毒起始位置 + 这个变量相对病毒起始位置的偏移 = 变量在被感染对象中的位置。

​ 是不是很好理解？

​ 继续思索:

​ 我们如何求病毒被插入感染对象后的起始位置，它是可变的。
​
​ 我们来看一下 call 指令，call 指令是将下一句指令的偏移压入堆栈，然后设置 eip 寄存器指向要跳转的地址。

​ 看以下代码:
​ 00401010 >/$ E8 00000000 ;call 00401015
​ 00401015 |$ 5B ;pop ebx
​
​ 就如这段代码 call 指令首先将下一句指令的地址 00401015h 压入堆栈，然后将 eip 寄存器指向 00401015h。

病毒技术中 call 指令详解:

​ 我们都知道 call 调用一个内存地址的话，编译器编译的是相对于调用地址的偏移 (注意：这里不是绝对偏移，相对偏移)，求调用地址的公式是调用地址 - (call 指令所处偏移 +5) = 相对偏移

​ 如上面这段代码，假设我们的代码是 call 00401016 的话，那么编译器会将机器码编译为 E8 00000001 。 E8 为 call 对应的机器码。 00000001 为偏移差，不懂自己根据上面的公式算算。

​ 我想大家看到这点应该明白我讲解 call 指令的作用了吧。。 因为 call 是将下句指令偏移压入堆栈，然后设置 eip 寄存器为调用地址。所以我们只要在调用地址中取得堆栈中 4 字节数据则为下句指令的偏移。

​ 这句代码执行的到 00401015 处通过 pop ebx 取得堆栈中 4 字节的数据 (这 4 字节数据则为 call 地址所处位置下句指令偏移地址，也就是 pop ebx 指令的地址)。通过这个地址我们就可以把它参照（就相当于上面所说的把它相当于病毒的起始地址），这样我们后面任何绝对地址均采用 这个地址 + 绝对地址相对这个地址的偏移 来取得真正的地址。说起来有些绕口，不过的确是这么个意思。初学者可以多读读理解下。

​ 这里很多不了解基础的人，写代码总是开始 E8 00000000

​ 其实你可以调用其他的地址，然后在其后写一些垃圾代码防止杀软的查杀。然后只要把下句指令偏移作为参照就行了。

​ 看代码。

​ szText db ‘Virus Dels Demo’, 0
​
​ __Entry:
​ call Dels2
​ Dels:
​ int 3
​ int 3
​ Dels2:
​ pop ebx
​ lea edx, [ebx + szText - Dels] ; edx = szText
​ ret
​

​ 代码很简单很好理解，通过调用 call dels, 然后通过 pop ebx 获得 dels 的偏移，然后通过 szText - Dels 获得 szText 变量基于 Dels 的相对偏移，然后在 + dels (ebx) = 重定位后的偏移。

​ 如果您能看到这里，我想您已经是基本理解了。其实就是这么简单，找一个能在运行时计算出来的地址，然后 + 上相对它的偏移则为重定位后的地址。主要思路就是这样，至于如何发挥就是大家的事情了，您可以发挥你的创造力来变形你的重定位代码，任何技术都是基础。。
